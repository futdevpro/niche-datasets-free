Buyer's Guide — Which Dataset for Which Workload?

Updated 2026-05-20 · 8 use cases · 20 datasets · 3 bundles

If you already know what you want to build, this page tells you exactly which dataset(s) to grab. If you don't — start with the workload that most matches yours; skip the others.

Use case 1 — Build an LLM pricing comparison page

"I want a table or chart comparing GPT vs Claude vs Llama by price/M tokens."

Primary pick: ai-models-pricing ($11). 802 (model × provider × variant) tuples from OpenRouter, with absolute costTierAbsolute enum (free / cheap / mid / premium / flagship), contextTier, uptimeTier, providerTier. The same model is in the table once per provider — so the buyer-question "where is Llama 3 cheapest" gets a one-query answer.

Don't bother with: huggingface-models (no pricing data, just downloads), ai-tools (high-level catalog, no per-token pricing).

Pro-tip — filter by semantic enum tier instead of writing thresholds

Every record carries derived enum-tier fields (full reference) so a query reads like the buyer's mental model. Example for use case 1 (LLM pricing comparison):

# cheap chat models with production-grade uptime, large context
jq '.[] | select(.useCaseTier == "chat"
              and .costTierAbsolute == "cheap"
              and .uptimeTier == "excellent"
              and .contextTier == "large")' \
  ai-models-pricing-sample.json

4 enum filters do what 4 inequality predicates + 5 OR-clauses for synonymous pipeline tags + a percentile cutoff that drifts every refresh would do otherwise. Same tier names work across related datasets: useCaseTier=code filters HF models + HF datasets + ai-models-pricing consistently.

Same patterns in pandas, Node.js fetch, and DuckDB SQL.
