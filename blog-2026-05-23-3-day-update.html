Published 2026-05-23 · Niche Datasets · Follow-up to the 13-Day Refresh Report (2026-05-20)

What the numbers say

npm did it again — 103.7% gross churn in three days

The 13-day report measured 122% gross churn. This 3-day window measured 103.7%. Both are over 100% — meaning more than half of the membership at the start of the window had been replaced by the end. The fact that two windows of very different lengths (13 days and 3 days) both land north of 100% is the real signal: the npm top-N ordering is not slowly drifting, it is continuously reshuffling. A snapshot taken at midnight does not describe the same population as a snapshot taken at noon. This is the dataset where pulling fresh, not pulling once, is the entire value proposition.

The "boring" club gained a member

The 13-day report had three datasets at literal zero adds/removes: design-resources, llmops-and-eval, vector-db-and-rag. In this 3-day window: four — design-resources, developer-tools, ai-agents, platform-engineering. The membership changed: llmops-and-eval and vector-db-and-rag both saw curator activity (+33 and +2 respectively), while developer-tools and ai-agents went quiet. If you are deciding refresh cadence per dataset, the membership of "boring" is itself worth watching: it tells you which awesome-list maintainers have moved on and which ones are still active.

llmops-and-eval finally moved

Zero adds in the 13-day window. +33 adds in this 3-day window. A single PR batch on one of the upstream awesome-lists can produce this kind of bursty pattern, and it is the reason a polling cadence beats a "decide once" cadence even for human-curated lists. Same logic applies to design-resources and platform-engineering — they are quiet right now, but a single merged PR can move 30 records in a day. The change-detection example is exactly the loop that catches these bursts.

HuggingFace datasets cooled off (briefly)

13-day window: 43.3% churn. 3-day window: 18.4%. The shorter window's lower gross rate suggests the source-cadence varies week-to-week — HF Hub publishing is bursty too, not steady-state. Pull-frequency for HF should still be high (daily is fine), but the per-pull yield is not constant.

HuggingFace models — 0 net, 89/89 churn

Exactly +89 added and −89 removed in 3 days, with 3,907 records updated. This is the "top-N by downloads" ranking pattern: the population size stays pinned (model is in or out by rank), and almost every record gets metadata-touched on every refresh (download counts, recent versions). Same pattern as npm — the value is in the freshness of the ranking, not the row count.

Cumulative 16-day view (since 2026-05-07 baseline)

Combining the 13-day window and this 3-day window, the catalog has grown 55,326 → 55,683 net (+357) since the 2026-05-07 baseline, with cumulative gross activity of roughly 9,000 additions, 60,000 metadata updates, and 8,300 removals across the 16 days. Two-thirds of all activity comes from three datasets: npm-packages, huggingface-models, and homebrew-packages. The other 17 datasets contribute the long tail of curator-driven incremental additions.

Pipeline notes

  • 0 validation errors across all 55,683 records through every refresh in the window — Zod schemas held cleanly. Same record as the 13-day report.
  • All 20 datasets refreshed in a single ~6-hour sweep on 2026-05-23 (21:03–23:02 CET), one dataset per ~15-minute tick. Per-dataset wall-clock: 0.6s to 3.4s, median under 2s.
  • Every refresh pushed a new sample to the free preview repo and pinged IndexNow (Bing / Yandex / Seznam / Naver) so the freshness badges show within minutes.
  • New since the 13-day report: bundle membership is now first-class in the schema — every dataset record advertises which themed bundles include it via Schema.org isPartOf and a flat bundles[] field in /datasets.json + per-dataset -meta.json. API docs · jq example.
  • Also new: liveRecordCount in /datasets.json ships the exact int from the live data file (versus recordCount, which is the marketing-rounded "X,XXX+" string). The numbers in the table above all come from liveRecordCount, not the marketing strings. Why the two fields.
