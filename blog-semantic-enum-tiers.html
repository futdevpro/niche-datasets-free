Semantic Enum Tiers — Filtering That Survives Refreshes and Crosses Datasets

2026-05-20 · Engineering notes from running 20 niche datasets in production

Every dataset in this catalog ships with extra enum fields that the source itself does not have — costTierAbsolute, useCaseTier, uptimeTier, popularityTier, recencyTier, licenseTier, and ~30 more. They look like noise next to the raw API fields. They are the most-used fields in every buyer query we've ever observed.

Why source-fields aren't enough

An npm package object has downloads: a number. A HuggingFace model has pipeline_tag: a string from a controlled vocabulary of ~30 entries. An OpenRouter endpoint has pricing.prompt: a USD-per-token decimal.

If you give these to a buyer, they cannot answer their own buyer-question. Specifically:

  • "Show me cheap models" — what's cheap? pricing.prompt < 0.000001 means nothing to a non-API-fluent human, and the threshold shifts every time a new flagship launches.
  • "Show me production-grade providers" — what's production-grade? uptime > 0.999? Why 3 nines vs 4? Where's the SLA mapping?
  • "Show me trending packages" — top 1%? Top 100? Top by downloads, by stars, by lifetime-vs-recent ratio?
  • "Show me chat models" — there are 5 different HuggingFace pipeline_tags that all mean "this is a chat assistant" — text-generation, text2text-generation, conversational, and edge cases.

The buyer wants buyer-aligned tiers with stable thresholds. We compute these once during the pipeline run and ship them as first-class fields.

Tier design rule #1 — absolute over percentile

Most catalogs that bother with tiering use percentile ranks: "this model is in the top-25% by downloads". That sounds rigorous and falls apart immediately for the buyer.

Percentile tier problem: If 4,000 new models launch on HuggingFace this week, every existing model drops a percentile bucket — even though nothing changed about it. "Top 5% by downloads" today and "top 25% by downloads" next month is the same model with the same downloads. The label moved because the cohort changed.

For most buyer-questions, an absolute threshold is more useful. If downloads > 1,000,000 = "viral", that label is stable across refreshes regardless of how the cohort shifts. Same model, same data, same label, week after week.

So we use percentile-ranks only for fields where "relative position" is genuinely what the buyer wants (e.g. popularityTier = "rest" vs "top25pct" vs "top5pct" vs "top1pct" — buyer-question is "is this a giant or a niche?", a percentile question). For everything else we hardcode absolute thresholds tied to real-world buyer meaning.

Worked example — ai-models-pricing has both

The ai-models-pricing dataset has two cost tiers, deliberately:

FieldTypeValuesBuyer-question it answers
costEfficiencyScorepercentilecheapest25pct / cheap50pct / midrange / expensive / unknown"How does this model compare to the rest of the catalog right now?"
costTierAbsoluteabsolutefree / cheap (<$1/M) / mid ($1-10/M) / premium ($10-50/M) / flagship (>$50/M)"Is this within my budget per million tokens?"

Both are useful, for different reasons. The percentile tier is comparative; the absolute tier is budgetary. A buyer running cost-modeling for production traffic only cares about the absolute one.

Tier design rule #2 — same enum across datasets when the buyer-question is the same

A buyer asking "show me chat models" wants the same answer whether they're querying huggingface-models, huggingface-datasets, or ai-models-pricing. Three different source schemas, three different controlled vocabularies for what "chat" means. The buyer doesn't care.

So useCaseTier is the same enum across all three:

chat
embedding
vision
audio
code
multimodal
nlp-classic     (token-classification / translation / fill-mask / etc.)
tabular         (huggingface-datasets only — models rarely target tabular)
other
unknown

Each dataset's extractor computes the field from its own source vocabulary, but the output enum is identical. A buyer running a single agent query across all three datasets can filter useCaseTier=code and get models, training datasets, and priced endpoints in one consistent result set.

Tier design rule #3 — name the absolute threshold after a real-world concept

Uptime is the cleanest example. The SLA industry has named the thresholds for 30 years:

Industry termuptimeTier valueThreshold
"four nines"excellent≥99.9%
"three nines"good99.0–99.89%
"two nines"acceptable95.0–98.99%
(below 2 nines)poor<95%
(no data)unknown—

A buyer doesn't have to learn our taxonomy. They look at uptimeTier=excellent and know exactly what it means, because it maps to the industry concept they already use to write SLAs.

What this looks like in a real query

From the free ai-models-pricing sample:

jq '
  .[]
  | select(.useCaseTier == "chat"
        and .costTierAbsolute == "cheap"
        and .uptimeTier == "excellent"
        and .contextTier == "large")
  | {name, provider, pricingPromptPerMillionUsd, contextLength}
' ai-models-pricing-sample.json

Reads exactly like the buyer's mental query: "give me a cheap chat model from a reliable provider with a big context window." 4 enum filters do what would otherwise take 4 inequality predicates with magic numbers + 5 OR-clauses for synonymous pipeline tags + a percentile cutoff that drifts every refresh.

Takeaway: the value of a catalog is not its row count. It's how easily the buyer can ask their own buyer-question against it without learning the schema. Semantic enum tiers — absolute where buyers think absolutely, percentile where they think relatively, same enum across datasets where the question is the same — are the single most-used filter in every workflow we've observed.
