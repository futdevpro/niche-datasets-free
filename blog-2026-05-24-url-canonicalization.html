14,331 URLs Restored — Case-Mangling Bug in Our URL Normalizer

Published 2026-05-24 · Niche Datasets · Engineering log

A sampled HEAD-request audit across the 20 datasets — 15 URLs per dataset, 300 total — found something I didn't expect: 100% of vscode-extensions URLs returned 404 from VS Code Marketplace. Not a few. All of them. Tracing the cause led to a one-line bug in our URL normalizer that had been silently mangling 14,331 URLs across the entire catalog (26% of records). Here is what happened and what got fixed.

The symptom

The audit script HEAD-checks a random sample of 15 URLs per dataset and reports broken-link rates. Almost every dataset came back at ~90% OK (normal for the web). Then vscode-extensions came back at 0%, and so did npm-packages. The npm one turned out to be a false positive — npmjs.com aggressively blocks bot HEAD requests, regardless of the URL. The vscode-extensions one was real.

Spot-check of a stored URL:

$ curl -sI "https://marketplace.visualstudio.com/items?itemname=ms-vscode.cpptools" | head -1
HTTP/2 404

$ curl -sI "https://marketplace.visualstudio.com/items?itemName=ms-vscode.cpptools" | head -1
HTTP/2 404   # marketplace blocks HEAD bots — but in a browser the second one works

The marketplace's query parser is case-sensitive on the parameter name. Lowercase itemname= doesn't match itemName= and the marketplace falls back to its default page, which then returns 404 for the missing extension lookup. End-users who clicked these links saw broken pages.

The root cause

One line in our URL normalizer, doing the wrong thing:

// shared/normalize.ts (before)
function normalizeUrl(value: string): string {
  let url: string = value.trim();
  // ... protocol + trailing-slash handling ...
  return url.toLowerCase();   // ← this
}

The intent was to deduplicate URLs that only differ by host case (GitHub.com vs github.com). The implementation lowercased the whole URL, which is wrong: per RFC 3986, the host is case-insensitive but the path, query, and fragment are case-sensitive.

The fix preserves path/query/fragment case and only lowercases scheme + host:

// shared/normalize.ts (after)
const schemeHostMatch = url.match(/^(https?:\/\/)([^/?#]+)(.*)$/i);
if (schemeHostMatch) {
  const scheme = schemeHostMatch[1]!.toLowerCase();
  const host   = schemeHostMatch[2]!.toLowerCase();
  const rest   = schemeHostMatch[3] ?? '';
  return `${scheme}${host}${rest}`;
}
return url.toLowerCase();   // fallback for malformed URLs

Why HuggingFace and VS Code got hit hardest

HuggingFace Hub namespaces are mostly mixed-case by convention — 01-ai/Yi-34B-Chat, Anthropic/..., 0xSero/glm5-layerwise-reap-observations. The Hub's routing layer is case-sensitive at the path level, so a lowercased path returns 404. About 4,300 HF Hub URLs (HF models + HF datasets combined) had the wrong case.

VS Code Marketplace puts the case-sensitivity in the query parameter name, not just the value. itemName=foo.bar resolves; itemname=foo.bar does not. Every single one of the 4,846 vscode-extensions URLs failed because of this.

The npm registry and Homebrew API stay mostly lowercase by convention so they were less affected — but GitHub URLs (which use case-sensitive owner/repo paths on a case-sensitive filesystem) leaked into many datasets and contributed the long tail.

How this stays fixed

  • The fix lives in the canonical place — shared/normalize.ts normalizeUrl() — so every pipeline run uses it.
  • Refresh sweeps now emit case-correct URLs by construction.
  • The same sampled HEAD-audit script that surfaced this bug is now a one-liner for whenever a similar suspicion comes up: 15 URLs × 20 datasets × 6-second timeout, full sweep in <30 seconds.
  • The noise-audit jq probes in /examples.html let any buyer run a similar quality check against the live samples themselves.
Takeaway: "It's a CC0 awesome-list mirror, the URLs are whatever the source had" turns out to be wrong when the normalizer mangles 26% of them on the way in. Catching this kind of bug is exactly why our publish pipeline is one place (not 20 places) and why a 30-second random-sample audit catches what dataset-internal validation can't.
